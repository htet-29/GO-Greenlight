# AGENTS.md - Greenlight API

## Project Overview

Go 1.25 REST API for managing movies and users. Uses `httprouter`, `pgx/v5` for PostgreSQL, and `sqlc` for type-safe SQL code generation.

## Commands

### Build & Run
```bash
go build -o ./tmp/main ./cmd/api/   # Build
./tmp/main                           # Run binary
air                                  # Dev with hot reload
```

### Test
```bash
go test ./...                        # Run all tests
go test ./cmd/api/                   # Test specific package
go test -run TestName ./...          # Run single test
go test -v ./...                     # Verbose output
```

### Lint & Validate
```bash
revive ./...                         # Lint (matches CI)
go vet ./...                         # Static analysis
go build -v ./...                    # Compile check
```

### Database Code Generation
```bash
sqlc generate                        # Generate Go from SQL queries
```

### CI Pipeline
GitHub Actions runs: `revive` → `go build -v ./...` → `go test -v ./...`

## Architecture

```
cmd/api/          # HTTP handlers, middleware, routes
internal/data/    # sqlc-generated DB code (DO NOT EDIT)
internal/domain/  # Business models and validation
internal/custom/  # Custom types (Runtime, Filters)
internal/validator/ # Validation utilities
internal/mailer/  # Email sending
queries/          # Raw SQL files for sqlc
migrations/       # Database schema migrations
```

## Code Style

### Imports
- Standard library first, then third-party, then internal packages
- Group imports with blank lines between groups
- Use aliases for collisions (e.g., `ht "html/template"`, `tt "text/template"`)

### Formatting
- Standard `gofmt` formatting (tabs for indentation)
- 1MB request body limit via `http.MaxBytesReader`
- JSON responses use `envelope` type: `map[string]any`

### Types & Naming
- **Domain models**: `internal/domain/` with JSON tags, validation functions as `ValidateXxx(v *validator.Validator, x *X)`
- **DB models**: `internal/data/` (auto-generated by sqlc, use `pgtype.Timestamptz`)
- **Custom types**: `internal/custom/` (e.g., `Runtime`, `Filters`, `Metadata`)
- Exported types use PascalCase, unexported use camelCase
- Handler methods: `app.{action}{Resource}Handler` (e.g., `createMovieHandler`)

### Error Handling
- Use `application` struct helper methods for HTTP errors:
  - `serverErrorResponse` - 500 internal errors
  - `badRequestResponse` - 400 malformed requests
  - `notFoundResponse` - 404 missing resources
  - `failedValidationResponse` - 422 validation errors
  - `editConflictResponse` - 409 version conflicts
- Log errors with `app.logError(r, err)` including method and URI
- Use `errors.Is` and `errors.As` for error type checking
- Database operations: check `sql.ErrNoRows` and `context.DeadlineExceeded`

### Context & Timeouts
- All database operations use 3-second context timeouts:
  ```go
  ctx, cancel := context.WithTimeout(r.Context(), 3*time.Second)
  defer cancel()
  ```
- Background tasks use `app.wg.Go()` with panic recovery via `app.background(fn)`

### Validation Pattern
```go
v := validator.New()
if domain.ValidateMovie(v, movie); !v.Valid() {
    app.failedValidationResponse(w, r, v.Errors)
    return
}
```

### Middleware Chain
Routes are wrapped in middleware (outer to inner): `recoverPanic → enableCORS → rateLimit → authenticate → handler`

### Database
- SQL queries in `queries/*.sql`, schema in `migrations/`
- Generated code in `internal/data/` via `sqlc.yaml`
- Connection pool: `pgxpool` with configurable max/min connections
- Use `data.New(pool)` to create query client

### Logging
- Structured logging with `log/slog` and text handler
- Log levels: `Info` for operations, `Error` for failures
- Include contextual attributes: `app.logger.Info("message", "key", value)`
